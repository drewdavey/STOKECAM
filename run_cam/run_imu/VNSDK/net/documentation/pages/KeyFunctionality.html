Key Functionality¶

Establishing Communication¶

Connecting to the Unit¶

It is first necessary to create a VNSDK::Sensor object, which we will use to send commands to the VectorNav unit.

Sensor sensor = new Sensor();

sensor = Sensor();

We can then connect to the VectorNav unit using the VNSDK::Sensor::AutoConnect() method.

sensor.AutoConnect(port);

sensor.AutoConnect(port_name);

Tip

If the baud rate is known, the VNSDK::Sensor::Connect() method can be used, which accepts an extra parameter of the expected baud rate. This is a quicker call, as it does not require scanning all the baud rates. However, this method does not perform any verification of connectivity, so it is necessary to call VNSDK::Sensor::VerifySensorConnectivity() to ensure communication.

Sensor sensor = new Sensor();
try
{
   sensor.Connect("COM8", Sensor.BaudRate.Baud115200);
}
catch (Exception ex)
{
   Console.WriteLine($"Error: Failed to connect to sensor. Reason: {ex.Message}");
   return;
}
if (!sensor.VerifySensorConnectivity())
{
      Console.WriteLine("Error: Failed to connect to unit.");
      return 1;
}

try
   sensor = Sensor();
   sensor.Connect('COM8', Sensor.BaudRate.(BaudRate+BaudRates).Baud115200);
catch ME
   fprintf('Error: Failed to connect to unit. Reason: %s\n', ME.message);
   return;
end
if ~sensor.VerifySensorConnectivity()
   disp('Error: Failed to connect to unit.');
   return;
end

If AutoConnect is used, no VerifySensorConnectivity call is necessary, as it verifies connectivity through the auto connection process at each baud rate. It is therefore the slower of the two methods, by default requiring a maximum of 4.5 seconds.

To maintain robustness while maintaining the speed of the Connect method, the AutoConnect can be used as a fallback when VerifySensorConnectivity fails.

Changing Baud Rate¶

Once communication has been established, the baud rate can be reconfigured at any point using the VNSDK::Sensor::ChangeBaudRate() method.

sensor.ChangeBaudRate(Sensor.BaudRate.Baud921600);

sensor.ChangeBaudRate(Registers.System.('BaudRate+BaudRates').Baud921600);

Note

The ChangeBaudRate method will send a command to the unit to reconfigure its baud rate, disconnect from serial, and reconnect to the serial port at the specified new baud rate.

Tip

To ensure the baud rate change persists over a unit power cycle, a VNSDK::Sensor::WriteSettings() command should be sent.

sensor.WriteSettings();

sensor.WriteSettings();

Disconnecting From the Unit¶

To disconnect from the unit, the VNSDK::Sensor::Disconnect() method can be called, which also closes the listening thread.

sensor.Disconnect();

sensor.Disconnect();  % We don't want to replace an object that already exists, and may keep the serial port locked

Accessing Registers¶

All configuration settings and many measurements are stored in one of 256 registers, which are accessible via synchronous polling and synchronous writing

Both measurement registers and configuration registers can be read, whereas only configuration registers can be written. This is accomplished best using the VNSDK::Sensor::ReadRegister() and VNSDK::Sensor::WriteRegister() methods, which are discussed in the following sections.

The fields in each Register object will match the fields listed in the VectorNav unit’s latest ICD, which can be downloaded directly from VectorNav’s website.

Note

The ReadRegister and WriteRegister methods are blocking calls. There is currently no mechanism for non-blocking calls.

Reading Registers¶

As discussed in Accessing Registers, the VNSDK::Sensor::ReadRegister() method is the preferred method to read registers. To do so, you must

  1. Create a Register object to be populated by the received response

  2. Call the ReadRegister method

  3. Access the received values in the original Register object

VNSDK.Registers.Attitude.YawPitchRoll ypr = new VNSDK.Registers.Attitude.YawPitchRoll();
// And then making an base class alias of the derived object to pass to the SDK API
// This is because .NET does not all polymorphism at the method parameter level
// And .NET does not allow casting the derived object as an argument in the function call
sensor.ReadRegister(ypr);
Console.WriteLine($"Current Reading: " +
    $"Yaw {ypr.yaw}, " +
    $"Pitch {ypr.pitch}, " +
    $"Roll {ypr.roll}");

ypr = Registers.Attitude.YawPitchRoll();
sensor.ReadRegister(ypr);
fprintf(['Current reading:\n Yaw: %f\t'...
    ' Pitch: %f\t Roll: %f\n'],ypr.yaw, ypr.pitch, ypr.roll);

Writing Registers¶

As discussed in Accessing Registers, the VNSDK::Sensor::writeRegister() method is the preferred method to read registers. To do so, you must

  1. Create a Register object

  2. Populate the Register object with the desired values to configure the unit

  3. Call the writeRegister method

// Set ASCII messages to YPR
VNSDK.Registers.System.AsyncOutputType asyncOutputType = new VNSDK.Registers.System.AsyncOutputType();
asyncOutputType.ador = VNSDK.Registers.System.AsyncOutputType.Ador.YPR;                       // YPR
asyncOutputType.serialPort = VNSDK.Registers.System.AsyncOutputType.SerialPort.ActiveSerial;       // Active Serial Port

sensor.WriteRegister(asyncOutputType);

% Set Measurements to YPR
asyncDataOutputType = Registers.System.AsyncOutputType();
asyncDataOutputType.ador = Registers.System.('AsyncOutputType+Ador').YPR;
asyncDataOutputType.serialPort = Registers.System.('AsyncOutputType+SerialPort').Serial1;
sensor.WriteRegister(asyncDataOutputType);

Reading Measurements¶

By default, all asynchronous measurements (except gnssSatInfo and gnssRawMeas), whether sourced from ASCII or binary outputs, are made available as a VNSDK::CompositeData object on the MeasurementQueue contained within the Sensor object once the packet is parsed.

The Composite Data Object¶

A CompositeData object contains each measurement group (e.g. Attitude) as public members, each of which is a structure containing each measurement type (e.g. Ypr). Exactly one CompositeData object is populated for each measurement message, each of which contains a subset of the possible measurement types. Because a measurement message likely does not contain each enabled measurement, the measurement type members are each a std::optional of the underlying datatypes defined in the unit’s ICD. .. Note

The CompositeData class does not contain the Common group. As such, any measurement types populated via a binary measurement in the Common group will show up in the underlying group. For example, a binary message which contains ypr from the Common group will be populated in the attitude member of a CompositeData object.

How to Read a Measurement¶

  1. Get a CompositeData object from the MeasurementQueue.

    This can be done in two ways:

    • Use VNSDK::Sensor::GetNextMeasurement(), which will block until the set timeout. If there is already a CompositeData object on the queue when called, it will return that object without blocking.

    • Use VNSDK::Sensor::GetMostRecentMeasurement()

    Both of these methods return a std::optional, which means that you must check to ensure that the returned value is populated using the has_value method.

  2. Validate that the measurement you want is in the VNSDK::CompositeData object. This can be done in two ways:

    • Use the VNSDK::CompositeData::matchesMessage() method

      This method checks to see which message populated this particular composite data object, using an exact match. If a composite data object matches a particular header or message, each included measurement type is guaranteed to have a value, so you can safely call value().

      This approach is especially useful if you are sending measurements via multiple output registers at once, and would like to take different actions for each one.

      The matchesMessage method can accept either a string literal in the case of ASCII messages, or the BinaryOutputRegister object we used earlier to configure the register in the case of a binary message.

    • Use the has_value method on each measurement you’re interested in.

      Each measurement type of the VNSDK::CompositeData object is a std::optional, so you can directly check if the value has been set by using has_value. This is most useful if you do not necessarily know or care from which message a particular measurement came, and would like to perform some action whenever a measurement is available.

Sending Commands¶

Each command able to be sent to the unit is available as a public method of the VNSDK::Sensor class using blocking calls.

To send a command, simply call the public method with the necessary parameters, if applicable. For example, to send a Set Initial Heading command, you can call

sensor.SetInitialHeading(45);

sensor.SetInitialHeading(45);

Error Handling¶

The .NET Sensor object throws exceptions when errors are encountered.

try
{
      sensor.AutoConnect(port);
}
catch (VnException e)
{
      // Do something with the exception
}

try
      sensor.AutoConnect(port_name);
catch e
      e.message
end
